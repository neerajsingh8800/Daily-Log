# 04: Query Optimization and Indexing

This module covers the core mechanics of database storage engines, how to dramatically accelerate data retrieval using **Indexing Strategies**, how to read database execution paths, and database scaling methodologies like **Partitioning and Sharding**.

---

## 1. Relational Storage Mechanics

To optimize queries, you must understand how a database engine reads data from disk into memory. Databases read data in fixed-size blocks called **Pages** (typically 8KB in PostgreSQL/SQL Server, 16KB in MySQL InnoDB).

### The Core Problem: Table Scans
*   **Sequential Scan (Seq Scan / Full Table Scan)**: The database engine reads *every single page* of a table from disk to find matching rows. If a table has 100 million rows spread across 1 million pages, a query filtering on a non-indexed column requires 1 million disk I/O reads.
*   **Index Scan**: The engine queries a highly optimized lookup tree structure to find the exact page pointers, reducing disk I/O operations from millions to single digits.

## 2. Advanced Indexing Strategies

An index is a powerful data structure that trades **Write Speed** and **Disk Space** for **Read Speed**. Every time you write data (`INSERT`/`UPDATE`/`DELETE`), the index must also be updated.

### 🏢 1. B-Tree Indexes (Balanced Tree)
The default index type in almost all relational databases. It keeps data sorted and allows search, sequential access, insertions, and deletions in logarithmic time.

*   **How it works**: It self-balances so that all leaf nodes are at the exact same depth.
*   **Time Complexity**: 
    $$O(\log_b N)$$ 
    *(Where $N$ is the number of rows and $b$ is the branching factor of the tree).*
*   **Best Used For**: Equality queries (`=`), range queries (`>`, `<`, `BETWEEN`), and sorting operations (`ORDER BY`).

#### Implementation (PostgreSQL / MySQL):
```sql
-- Creating a standard B-Tree index on an email column
CREATE INDEX idx_users_email ON users(email);
```
### 🧩 2. Composite Indexes (Multi-Column)

An index built on multiple columns simultaneously. The order of the columns in the index creation script matters completely due to the Left-to-Right Prefix Rule.

Left-to-Right Prefix Rule: An index on (last_name, first_name) can speed up queries looking for last_name, or last_name AND first_name. However, it cannot optimize a query looking only for first_name.
```sql
-- Ideal for queries filtering by both department and status
CREATE INDEX idx_emp_dept_status ON employees(department_id, performance_status);
```
###  🏁 3. Bitmap Indexes

Instead of storing direct row pointers, a bitmap index uses strings of 0s and 1s (bit arrays) to represent whether a row matches a specific value.

Best Used For: Columns with Low Cardinality (columns with very few unique values, like gender, marital_status, or boolean_flags) in analytical data warehouses (OLAP).

Why not OLTP?: Locking a single bit during a write operation frequently locks surrounding records, stalling high-concurrency transactional writes.

## 3. Query Execution Plans

Before optimizing an inefficient query, you must ask the database engine to show you its structural execution roadmap using the EXPLAIN statement.

### The Optimization Keywords:
EXPLAIN: Shows the execution plan and the estimated computational costs generated by the database optimizer without running the query.

EXPLAIN ANALYZE: Actually executes the query in real-time, providing both the optimizer's estimations and the actual time metrics.

### Analyzing Database Output Logs

Consider this slow query executing across an unindexed logging table:
```sql
EXPLAIN ANALYZE 
SELECT user_id, count(*) 
FROM application_logs 
WHERE status_code = 'ERROR'
```

## 4. Horizontal Data Scaling: Partitioning vs. Sharding

When a database grows so massive that indexes no longer fit into RAM, you must scale your data layer horizontally.

### 🍕 Database Partitioning (Logical Division)

Partitioning breaks down a single massive table into smaller, more manageable logical sub-tables within the exact same database instance.

Range Partitioning: Splitting data based on specific value brackets (e.g., partitioning an orders table by transaction date: Partition_2025, Partition_2026).

List Partitioning: Splitting data based on an explicit array of string constants (e.g., partitioning by region: US, EU, APAC).

Hash Partitioning: Applying a mathematical hash algorithm to a routing column key to evenly distribute rows across an established number of logical partitions.

### 🌐 Database Sharding (Physical Distribution)

Sharding is an architecture that distributes rows of a table across completely separate physical server instances and network locations. Each machine is called a Shard.

Shared-Nothing Architecture: Shards do not share memory or computing power.

The Routing Key: A designated column (e.g., Tenant_ID or User_ID) dictates which physical shard handles the write/read traffic.
